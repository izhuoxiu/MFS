# 1. 项目背景

## 1.1 人类感官的缺失：为什么需要标准触觉语言

触觉是人类最早发育、最直接的身体感知通道，但在数字世界中，它长期被忽视。我们通过屏幕“看”到内容，通过扬声器“听”到声音，却极少通过震动“感受”到信息。

随着手机线性马达、穿戴设备、VR/AR 控制器、智能座舱、触觉手套等硬件普及，触觉反馈的硬件基础已经成熟。然而，**软件层面缺乏一套统一的描述语言**——每个平台（iOS、Android、Windows）各有自己的 API，开发者不得不为每个硬件重复编写波形代码，内容创作者也无法像撰写字幕一样“撰写触觉”。

此外，传统的触觉反馈多为生硬的系统提示，未能与环境和多模态信息深度融合。新一代交互要求触觉能够对**声音、视频、动作、文字、表情、情绪、空间、位置**等全方位多模态要素做出精准、细腻的触觉级实时反馈，这迫切需要一套高内聚、低耦合的标准协议。

## 1.2 当前行业痛点

| 痛点 | 说明 |
| :--- | :--- |
| **硬件碎片化** | 不同厂商的马达（如 ERM、LRA、压电马达）特性差异巨大，同样的震动参数在不同手机上效果天差地别 |
| **语义缺失** | 现有 API 只提供“频率、振幅、时长”等底层参数，没有“心跳”、“上升”、“恐惧传递”等高阶概念 |
| **内容不可复用** | 一段精心设计的触觉脚本无法从一个 App 迁移到另一个，更无法跨平台、跨设备终端复用 |
| **创作门槛高** | 触觉设计需要专业波形工具和硬件调试，普通内容创作者、UI设计师或文本创作者无法参与 |
| **模态融合断层** | 触觉与声音、视频、空间物理位置及人类情感表情之间缺乏统一的映射逻辑与同步标准 |

## 1.3 本规范的目标与适用范围

**Haptic Event Schema (HES)** 是一套开放、免费、硬件无关的多模态触觉描述标准。它定义：

- 一套基础触觉词汇（从“轻触”到“心跳”，涵盖力量、节奏、空间、位置、运动、情绪等维度）
- 一种通用脚本格式（HSF，类似 MIDI 或 SRT，支持多轨动态扩展）
- 一套全模态内容（文本、视音频、表情、空间物理状态）到触觉的映射指南

**适用范围**：移动设备、游戏手柄、智能穿戴设备（手套、背心）、VR/AR/MR 控制器、汽车智能座舱触觉提示、无障碍辅助设备等一切具备震动反馈能力的终端。

# 2. 设计原则

## 2.1 硬件无关性 (Hardware Agnostic)

HES 只描述“应该感受到什么”（What to feel），而不规定“具体怎么震动”（How to vibrate）。具体波形渲染由各平台的实现层（Renderer）根据本地马达硬件特性（如共振频率、响应时间）进行动态最优化适配与渲染。

## 2.2 语义优先于波形 (Semantic over Waveform)

开发者优先使用高级语义标签 `Rhythm.Heartbeat` 或 `Emotion.Tension`，而非 `频率=60Hz, 振幅=0.8, 时长=200ms`。语义层保证了跨设备体验的主观一致性，使触觉具备了可读性与编辑性。

## 2.3 组合性与扩展性 (Composable & Extensible)

基础词条可以组合成复合触觉（如“空间接近 + 情绪紧张”）。同时允许开发者自定义扩展词条（保留私有编码区），并支持通过参数化包络（Envelope）对基础词条进行二次微调。

## 2.4 向后兼容 (Backward Compatibility)

新版本词典必须保持旧编码的有效性。不支持高级空间感或情绪感词条的低端硬件设备应具备智能降级策略（如自动替换为基础的 `Tap.Light` 或选择性静默）。

## 2.5 多模态对齐性 (Multimodal Alignment)

触觉事件必须能够与声音的时域/频域特征、视频的光流/色彩突变、文字的语义情感倾向、表情与情绪的强弱变化，以及空间三维位置的变化保持微秒级的精准同步与线性/非线性映射。

# 3. 基础触觉分类体系 (Taxonomy)

我们将人类可感知的触觉以及多模态触发源（声音、视频、动作、文字、表情、情绪、空间、位置）归纳并映射为 **5 大维度**，共 **18 个子类**。

## 3.1 力量感 (Force) —— 震动的“重量”与“锐度”

常用于映射物理碰撞、动作打击、视频中的爆炸、文字中的强语气表达。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 轻触 | Tap | 蜻蜓点水，极短促，常用于文字按键、UI微交互 |
| 敲击 | Knock | 有明确起止的撞击感，对应普通步态、常规鼓点 |
| 重击 | Heavy | 沉闷、有力、下坠感，对应重物落地、重拳动作 |
| 爆发 | Burst | 瞬间释放的巨大能量，对应视频爆炸、闪光、情绪破防 |
| 震荡 | Oscillate | 持续高频的颤抖，对应引擎轰鸣、声乐长音、持续警报 |
| 碾压 | Crush | 逐渐加重直至极限，对应骨骼碎裂、机械挤压、极度压抑文本 |

## 3.2 节奏感 (Rhythm) —— 震动的“时间排列”

常用于映射音频BPM、呼吸节奏、动作频次、心率变化。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 单次 | Single | 孤立的、一次性的触觉独立单元 |
| 重复 | Repeat | 均匀的来回往复，如钟表滴答、规则机械运转 |
| 脉冲 | Pulse | 规律且带有间隙的推进，对应雷达扫描、警示闪烁 |
| 心跳 | Heartbeat | 强弱交替的“咚哒”模式，直接映射生物心率与紧张表情 |
| 波浪 | Wave | 强度呈正弦曲线起伏，对应呼吸感、水流涌动、抒情音乐 |
| 滚奏 | Roll | 极快速密集的连续敲击，对应机枪射击、鼓点SOLO、动作疾跑 |

## 3.3 空间感与位置感 (Space & Location) —— 震动的“方位、距离与轨迹”

用于对空间三维物理位置、绝对/相对坐标、视频运动镜头及三维音频做出位置级反馈。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 接近 | Approach | 由弱变强。映射物体或光源靠近、距离坐标缩短 |
| 远离 | Retreat | 由强变弱。映射物体离去、距离坐标拉远 |
| 穿过 | Through | 从左到右或从前到后的位移感，映射跨渠道/多点马达传导 |
| 环绕 | Orbit | 围绕身体/设备旋转，多马达协同产生方位角交替变化 |
| 扩散 | Expand | 从中心向四周散开，映射波纹涟漪、视野张开、爆炸冲击波 |
| 聚集 | Contract | 从四周向中心收拢，映射黑洞吸入、视线聚焦、位置坍缩 |

> *注：空间感与位置感需要多马达矩阵或智能穿戴设备（如触觉背心）支持，单马达设备上将降级为宏观的振幅渐变。*

## 3.4 运动感 (Motion) —— 震动的“物理惯性与动作特征”

用于精准对齐物理动作行为、视频运镜速度（如推拉摇移）、动态手势及文字语速变化。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 上升 | Rise | 频率或强度递增，对应跳跃起飞、情绪激昂、音调拉高 |
| 下降 | Fall | 频率或强度递减，对应失重坠落、画面俯冲、音调沉降 |
| 加速 | Accel | 间隔越来越短，对应奔跑加速、心率飙升、动作节奏紧凑 |
| 减速 | Decel | 间隔越来越长，对应汽车刹车缓冲、动作收尾、语速放缓 |
| 悬浮 | Hover | 极度轻微且持续的飘忽感，对应零重力状态、微风拂面、平静表情 |
| 急停 | Brake | 高速震动瞬间归零，对应动作硬直、突发碰撞、情绪顿悟 |

## 3.5 情绪感与表情感 (Emotion & Expression) —— 震动的“心理暗示与生理微反应”

用于映射文本情感色彩（悲伤、愤怒、喜悦等）、面部表情微表情编码（FACS）、视频氛围及音频调性。

| 子类 | 英文 | 描述 |
| :--- | :--- | :--- |
| 紧张 | Tension | 高频细碎、不规律，对应焦虑表情、悬疑文字、心跳加速 |
| 压迫 | Oppress | 沉重、喘不过气，对应愤怒面容、巨兽凝视、低频声环境 |
| 释放 | Release | 紧张后的突然松弛，对应叹气、大笑、危机解除、大调和弦 |
| 兴奋 | Excite | 高频率、高强度的连续刺激，对应狂喜、尖叫、高能打斗画面 |
| 平静 | Calm | 极低频、极柔和的包裹感，对应微笑表情、舒缓文本、自然风光 |
| 诡异 | Weird | 不和谐的非对称、非线性震动，对应惊悚表情、反常动作、恐怖氛围 |

# 4. 事件编码规范

## 4.1 编码规则

每个标准事件由 **前缀字母 + 三位数字** 组成，前缀代表所属维度：

| 前缀 | 维度 | 示例 | 映射主要模态数据源 |
| :--- | :--- | :--- | :--- |
| F | 力量感 (Force) | F-101 | 声音振幅、物理碰撞、视频撞击、动作力度 |
| R | 节奏感 (Rhythm) | R-104 | 音频BPM、心率变化、动作频次、语言节奏 |
| S | 空间感与位置感 (Space) | S-201 | 3D空间坐标、绝对/相对位置、视频运镜、双声道音频相位 |
| M | 运动感 (Motion) | M-301 | 动作速度矢量、视频光流、物理惯性、语速 |
| E | 情绪感与表情感 (Emotion)| E-401 | 文本情感得分、面部表情特征点、画面冷暖氛围 |

数字段从 001 开始。为了保证扩展性，规范规定：**000–099 为私有扩展/自定义区**（保留给各厂商开发者自由定义特殊硬件波形）；**100–999 为公开标准词条**（遵循 HES 标准规范）。

## 4.2 事件属性

每个词条在 Schema 定义中应至少包含以下属性：

| 属性 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | string | 唯一编码，标准词条必须以公开标准区间（100-999）命名，如 `F-101` |
| `name` | string | 标准英文名，采用点分法表示层级，如 `Tap.Light` |
| `cn_name` | string | 中文名，如 `轻触` |
| `category` | string | 所属维度，取值为 `Force`, `Rhythm`, `Space`, `Motion`, `Emotion` |
| `intensity_default` | float (0–1) | 默认强度基准值 |
| `duration_ms` | int | 建议持续时间（毫秒），-1 表示由关联源动态决定持续时间 |
| `waveform_hint` | string | 波形几何及包络建议（方波/正弦波/锯齿波/白噪波/自定义包络） |
| `description` | string | 物理及生理心理感受的详细文本描述 |

# 6. 触觉脚本格式 (HSF) 规范

## 6.1 文件结构

HSF 使用 JSON 作为载体，具备多轨扩展性，支持针对不同物理位置的硬件（例如左侧马达、右侧马达、头显马达）进行单独分轨。

```json
{
  "schema": "HapticScriptFormat",
  "version": "0.1",
  "meta": {
    "title": "捉秀示例多模态联动脚本",
    "duration_ms": 32500,
    "bpm": 113,
    "author": "Pulse Project Team",
    "multimodal_source": {
      "audio_track_ref": "audio_001.mp3",
      "video_track_ref": "video_001.mp4",
      "text_track_ref": "subtitle_001.srt"
    }
  },
  "tracks": [
    {
      "id": "left_hand_motor",
      "device_target": "controller_left",
      "location_tag": "hand_left",
      "events": [
        {
          "id": "evt_001",
          "time_ms": 15000,
          "duration_ms": 40,
          "event_ref": "F-102",
          "intensity": 0.6,
          "note": "对应动作：左手挥拳打击 & 视频闪烁",
          "envelope": {
            "attack_ms": 5,
            "decay_ms": 30,
            "release_ms": 5
          }
        }
      ]
    },
    {
      "id": "right_hand_motor",
      "device_target": "controller_right",
      "location_tag": "hand_right",
      "events": [
        {
          "id": "evt_002",
          "time_ms": 15080,
          "duration_ms": 80,
          "event_ref": "F-103",
          "intensity": 0.8,
          "note": "对应动作：右手重击重音对齐"
        }
      ]
    }
  ]
}

```
## 6.2 字段定义
| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| schema | string | 是 | 固定为 "HapticScriptFormat" |
| version | string | 是 | 版本号，当前为 "0.1" |
| meta | object | 否 | 元数据，包含标题、总时长、BPM及多模态参考源信息 |
| tracks | array | 是 | 轨道列表，每个轨道对应一个物理硬件位置或逻辑触觉通道 |
| track.id | string | 是 | 轨道唯一标识符 |
| track.device_target | string | 否 | 设备类型提示（如 phone_motor, vest_chest, glove_finger） |
| track.location_tag | string | 否 | 触觉物理空间位置标签（如 hand_left, hand_right, head, waist） |
| track.events | array | 是 | 事件列表，必须在轨道内按照时间戳升序严格排列 |

| event.event_ref | string | 是 | 标准事件词典编码（如 F-101, E-404 等） |
| event.intensity | float | 否 | 强度缩放系数（0.0–1.0），用于对词典默认强度进行二次增益调节 |
| event.note | string | 否 | 供编辑、标注使用的描述性备注（如多模态对齐提示） |

| envelope.attack_ms | int | 否 | 起音时间（毫秒），控制震动达到峰值的平滑度 |
| envelope.decay_ms | int | 否 | 衰减时间（毫秒），从峰值过渡到稳定阶段的时间 |
| envelope.release_ms | int | 否 | 释放时间（毫秒），从震动收尾到静默的时间 |
## 6.3 排序与合并法则
 * **严格时间序**：所有轨道内部的事件对象必须严格按照 time_ms 进行升序排列，以便于流式渲染引擎高效读取和降低寻址开销。
 * **重叠混合策略（Mixing Rule）**：当同一轨道在同一时间段内出现两个或多个 HES 事件重叠时，系统渲染器默认遵循**“强度取大（Max Amplitude Principle）+ 频率混叠（Frequency Mixing）”**原则，也可通过自定义包络参数选择直接覆盖或线性叠加。
 * **硬件保护限流**：事件触发频率建议单机单马达设备上**不超过 8 次/秒**。高密度的滚奏（R-106）在渲染层需受限于硬件的响应阈值，以保护马达寿命并防止用户产生生理麻木感。

| **列表滑动选择** | 列表滚动越过一个 Item 节点（Tick） | R-105 (Wave) 或 F-101 | 模拟机械齿轮的拨动感，轻微而致密 |
| **界面长按 (Long Press)** | 触屏长按停留时长 > 500ms | F-106 (Crush) | 模拟能量聚集与下压感，随时间加重 |
| **通知高能到达** | 系统核心高优先级消息推送 | R-104 (Heartbeat) | 模拟人体生理心跳，以不突兀的形式吸引高关注度 |

## 7.5 冲突解决与仲裁法则
当多个源（如背景音频、突发视频爆炸、用户实时 UI 交互点击）同时产生 HES 事件请求时，渲染引擎必须遵循以下仲裁机制：
 * **优先级排序**：**用户实时交互事件（UI 操作/动作直接反馈） > 突发剧烈视觉事件（爆炸/闪白） > 音频主干特征（底鼓/高潮） > 背景氛围情绪（情绪感/平静等）**。
 * **动态限流合并**：若高优先级的 UI点击 与 音频底鼓 同时发生，取两者强度最高者，并允许进行频率混叠，但必须对叠加后的最大输出振幅实施防削波（Anti-clipping）限幅处理，避免烧毁马达或产生硬件破音。
# 8. 开发者实现规范
## 8.1 解析器（Parser）实现要求
 * **完备的 JSON 校验**：解析器必须能够支持标准 JSON 格式的 HSF 文本流或二进制流的解析，对非法字段、嵌套错误需具备完备的容错捕获机制。
 * **版本向前兼容与识别**：必须严格读取并解析 version 字段。若遇到高版本 HSF 文件（如 v0.2），当前 v0.1 解析器必须能够忽略不支持的高级参数，并正确提取基础事件字段。
 * **未知事件防崩溃（Fallback Map）**：对于任何自定义或当前版本不支持的 event_ref（如无法识别的空间感编码），解析器不得报错中断，必须将其统一映射到通用降级表（详见 8.4）。
## 8.2 渲染器最小实现（Android / iOS）
各平台渲染实现层负责将抽象的 HES 事件实时翻译为系统原生的触觉 API。
| 平台 | 基础映射逻辑 | 降级策略实现 |
|---|---|---|
| **Android** | 将标准 event_ref 的时间、强度及包络参数实时转换为原生的 android.os.VibrationEffect 对象，并通过 Vibrator 服务执行输出。 | 当设备不支持振幅/频率控制（如旧式 ERM 转子马达）时，所有不支持事件一律退化为：VibrationEffect.createOneShot(50, VibrationEffect.DEFAULT_AMPLITUDE) |
| **iOS** | 利用 CoreHaptics 框架，将 HES 事件及其属性动态构造为 CHHapticPattern 字典，并创建 CHHapticPatternPlayer 进行实时或预加载播放。 | 当运行在不支持 CoreHaptics 的旧款 iPhone 设备（如 iPhone 6s 之前）时，自动退化为调用传统的系统反馈：UIImpactFeedbackGenerator(style: .medium).impactOccurred() |
## 8.3 性能与时延要求
触觉对于时延的敏感度极高，必须保证“视听触”高度同频。
 * **解析开销**：单次解析 HSF 文件或段落的时间必须 **≤ 5ms**（在移动端主流 CPU 基准下，针对 10 分钟或 2000 个事件长度的脚本）。
 * **实时调度时延**：从接收到事件到调用系统 API 发出震动指令的软件调度时延必须 **小于 10ms**。
 * **CPU 实时渲染占用**：持续高频播放触觉脚本时，触觉引擎线程的 CPU 占用率必须 **< 3%**。
 * **内存开销**：常驻内存缓冲区的静态词典与动态事件队列总开销必须 **< 20MB**。
## 8.4 智能降级策略（Fallback Strategy）
 * **硬件基础能力不支持**：若目标终端设备仅搭载了传统的非线性马达，无法精细控制频率和振幅，则渲染引擎应将所有复杂的空间、情绪、运动事件，自动精简并替换为标准的 F-101 (Tap.Light) 或固定时长的短震。
 * **CPU 系统过载保护**：当检测到终端 CPU 负载过高或线程排队严重时，渲染引擎应自动启动限频机制，通过时间窗口算法合并 **50ms** 内相邻的、同维度的低优先级触觉事件。
 * **马达物理过热保护**：由于长时间的高强度、长波形震动会导致马达线圈发热，渲染层应建立虚拟温度追踪模型，当累计满载震动时间触发阈值时，自动等比例等差调低 intensity 系数（如降低 30% 输出），甚至短暂进入静默保护状态，待冷却指标恢复后继续正常输出。
# 9. 应用场景
## 9.1 短视频与音乐流媒体（捉秀 App 核心载体）
通过精准标注或 AI 自动生成的 HES 触觉脚本，让用户在刷短视频或收听电音、摇滚、古典等音乐时，手机能够随音浪节奏、画面高能瞬态同步释放细腻、富有层次感的触觉震动，开创“视、听、触”三位一体的全新多模态沉浸娱乐时代。
## 9.2 无障碍辅助感知
为视障或听障人群提供开创性的多模态辅助导航与信息感知。将周围环境的物理障碍距离（空间接近 S-201 / 远离 S-202）、路线转弯导航指示、周围人群的面部情绪和表情特征等，实时翻译为特定的编码触觉，通过可穿戴触觉设备传递给视听障用户。
## 3. 游戏互动深度反馈
将游戏内复杂多变的环境事件解耦。主角的脚步声动作、不同枪械的后坐力力量感、远处的突发爆炸、敌人的空间接近方位坐标，皆可通过 HSF 轨道脚本无缝映射至手柄或震动背心中，大幅提升核心玩家的游戏临场感。
## 9.4 智能座舱与驾驶安全预警
在汽车智能座舱中，当车辆遭遇车道偏离、盲区障碍物快速接近、前车急刹等紧急物理状况时，系统不再单靠刺耳的警报声，而是通过方向盘、座椅的多点多马达矩阵输出高强度的空间运动触觉（如 M-306 急停 + E-401 紧张），实现比听觉更直接、人类生理反应更迅速的安全预警。
## 9.5 通知信息语义化（无感通知）
改变千篇一律的系统默认震动。不同的社交软件、不同的人物联系人（如紧急联系人与普通群聊）、不同的通知语义类型（微信重要提及、日历闹钟、邮件送达），均可被分配专属且具备明显心理暗示的 HES 事件词条。用户无需点亮和观看屏幕，仅凭身体接触即可瞬间获知消息的重要程度。

| **HSF** | Haptic Script Format | **触觉脚本格式**。基于 JSON 结构设计的多轨制触觉描述文件规范，作为承载 HES 事件的核心文本载体。 |
| **事件 (Event)** | Haptic Event | 触觉描述的最小独立单元，具备时间戳、持续时长、强度、标准词典编码等关键属性。 |
| **轨道 (Track)** | Haptic Track | 时间轴上的多模态事件集合序列，通常一条轨道唯一绑定并映射一个物理硬件触觉通道。 |
| **包络 (Envelope)** | Waveform Envelope | 指震动信号在物理时域中随时间演变的强度外轮廓，通常由起音(Attack)、衰减(Decay)、维持(Sustain)、释放(Release)四大核心阶段的参数化控制构成。 |
| **渲染器 (Renderer)** | Haptic Renderer | 驻留在系统底层或 SDK 内部的运行渲染模块，负责解析 HSF 文件，并将抽象的 HES 语义编码高效转化为具体平台硬件直接执行的波形控制数据流。 |
| **降级 (Fallback)** | Haptic Fallback Mechanism | 当执行终端面临硬件不兼容、系统 CPU 过载、或者马达物理过热等异常约束时，主动降级采用通用、替代的高安全系数轻量触觉策略的容错算法流程。 |
